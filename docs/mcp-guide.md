# MCP 使用指南

Trace UI 内置 MCP（Model Context Protocol）Server，将完整的 trace 分析引擎以标准化接口暴露给 AI 工具。

这意味着你可以用自然语言告诉 AI 你的分析目标，AI 会自主调用 10 个分析工具完成操作——打开 trace 文件、搜索指令、执行污点分析、查询调用树、读取内存、识别密码算法——不需要你手动操作界面。

**能力边界说明**：MCP 提供的 10 个工具返回的数据（搜索结果、污点分析、调用树、内存内容等）是**确定性的、准确的**——它们和你在 GUI 中看到的结果完全一致。但 AI 基于这些数据做出的**推理和解释**（如"这段代码在做 AES 密钥扩展"、"这个函数是签名校验逻辑"）属于 AI 的分析判断，可能存在误判，尤其在涉及复杂 ARM64 计算链、算法识别上下文判断等场景时。建议将 AI 的分析结论作为**辅助参考**，结合你的逆向经验做最终判断。

本文档覆盖：
- MCP 相比 GUI 的优势
- 从零配置到第一次分析的完整流程
- 4 个典型逆向分析场景的提示词和工作流
- 10 个 MCP 工具的详细参数和使用说明

## MCP vs GUI：为什么要用 AI 分析 trace？

Trace UI 的 GUI 提供了完整的可视化分析能力，但在以下场景中，MCP + AI 有明显优势：

| 场景 | GUI 操作 | MCP + AI |
|------|---------|----------|
| 分析一个加密函数的数据流 | 手动搜索关键字 → 右键配置污点源 → 切换过滤模式逐行查看结果 → 手动构建依赖图 | 一句提示词，AI 自动完成搜索 + 污点分析，并用自然语言解释数据流 |
| 在百万行 trace 中找关键函数 | 在调用树中逐层展开、双击跳转、反复折叠/展开 | AI 自动遍历调用树，按你的描述（如"找到所有调用了 JNI 函数的节点"）筛选 |
| 批量分析多个 trace 文件 | 逐个打开文件、手动分析、记录结果 | AI 可以批量打开多个 trace，逐个分析并汇总对比报告 |
| 理解一段复杂计算逻辑 | 查看寄存器值、手动推导计算过程 | AI 读取完整寄存器数据和污点结果，尝试用自然语言解释计算过程（复杂逻辑可能需要人工验证） |
| 快速定位加密算法 | 菜单触发扫描 → 查看结果列表 → 逐个跳转确认 | AI 扫描后自动分析每个匹配的上下文，辅助判断是否为真正的加密调用（需人工确认） |

### 核心优势

- **自动化串联**：AI 自主串联多个工具完成复杂分析流程，你只需描述目标
- **自然语言交互**：不需要记住操作步骤和快捷键，用中文描述分析意图即可
- **批量处理**：一次分析多个 trace 文件，自动对比差异
- **智能解读**：AI 不只返回原始数据，还能尝试解释含义、识别模式、给出逆向建议（AI 的推理结论仅供参考，建议结合经验判断）
- **可复现**：分析流程以提示词形式保存，下次分析类似 trace 直接复用

### 什么时候用 GUI 更好？

- **可视化浏览**：需要直观查看 trace 流、折叠/展开函数体、拖动 Minimap 导航
- **交互式标注**：高亮、注释、删除线等标记需要视觉反馈
- **DEF/USE 箭头**：可视化寄存器值在指令间的传播路径
- **实时调试**：需要在指令间来回跳转、快速切换查看不同寄存器和内存

**最佳实践**：GUI 用于交互式浏览和标注，MCP + AI 用于自动化分析和批量处理。两者可以同时使用——桌面应用内置的 HTTP MCP 服务与 GUI 共享会话状态。

## 快速开始

### 第一步：编译 trace-cli

```bash
# 仅编译 trace-cli
./build.sh cli

# 或者编译完整项目（桌面应用 + trace-cli）
./build.sh release
```

编译完成后，`trace-cli` 二进制位于 `target/release/trace-cli`。

### 第二步：配置 MCP

**方式一：独立 CLI（推荐）**

将 `trace-cli` 注册为 MCP Server：

```bash
# Claude Code
claude mcp add trace-ui -- /path/to/trace-cli

# Claude Desktop（编辑 claude_desktop_config.json）
{
  "mcpServers": {
    "trace-ui": { "command": "/path/to/trace-cli" }
  }
}
```

**方式二：桌面应用内置 HTTP 服务**

在 Trace UI 桌面应用中启动 MCP Server（Settings → Preferences → MCP），然后配置 AI 客户端：

```bash
claude mcp add trace-ui --transport http http://127.0.0.1:19821/mcp
```

### 第三步：验证连接

在 AI 客户端中输入：

```
打开一个 trace 文件，告诉我基本信息
```

AI 应该调用 `open_trace` 并返回 trace 的概览信息（模块名、入口地址、格式、函数数量、总行数等）。

### 第四步：第一次分析

```
打开 /path/to/your/trace.log 文件，告诉我这个 trace 有多少行指令，
然后扫描是否存在已知的密码算法。
```

AI 会依次调用：
1. `open_trace` — 打开文件并构建索引
2. `analyze_crypto` — 扫描密码算法常量，并返回匹配位置的上下文指令

你会看到 AI 返回 trace 的基本信息和检测到的算法列表。

## 实战场景

### 场景一：全量自动化分析

**适用情况**：拿到一个新的 trace 文件，想要快速了解整体情况。

**提示词：**

```
打开 /path/to/trace.log，对这个 trace 做一次全面分析：

1. 基本信息：总行数、格式
2. 函数调用概览：有哪些函数被调用，调用次数最多的是哪些
3. 密码算法扫描：是否检测到已知算法
4. 字符串提取：有哪些有意义的运行时字符串（过滤掉太短的）
5. 基于以上信息，给出你的分析建议：哪些函数值得深入分析，可能的加密/签名逻辑在哪里
```

**AI 内部工具调用流程：**
1. `open_trace` → 获取基本信息（session_id、模块名、入口地址、格式、函数数量、总行数）
2. `analyze_function` （无参数）→ 获取函数调用聚合列表
3. `analyze_crypto` → 扫描密码算法，返回匹配结果及上下文
4. `get_strings(min_len=6)` → 提取有意义的字符串
5. 综合分析，输出报告和建议（步骤 1-4 返回的数据是准确的，第 5 步的分析建议是 AI 推理结果，仅供参考）

**GUI 下的等效操作：** 手动打开文件 → 等待索引 → 展开调用树逐层查看 → 菜单触发密码扫描 → 切换到字符串面板调整过滤 → 自己总结分析。MCP 将数据采集步骤压缩为一句提示词，但最终的分析判断仍需结合逆向经验。

### 场景二：追踪加密/签名函数

**适用情况**：已知 trace 中存在加密或签名操作，需要追踪其输入输出数据流。

**提示词：**

```
打开 /path/to/trace.log，帮我分析加密流程：

1. 先扫描密码算法常量，看看 trace 中用了什么加密算法
2. 根据扫描到的算法匹配位置，找到这些指令所在的函数（通过调用树定位）
3. 对该函数入口处的输入寄存器（通常是 X0-X3）做污点分析，追踪数据来源
4. 分析加密输入是如何被计算出来的
```

**AI 内部工具调用流程：**
1. `open_trace` → 获取基本信息
2. `analyze_crypto` → 识别算法类型和匹配位置（已包含上下文指令）
3. `get_call_tree` → 通过调用树定位该指令所属的函数
4. `analyze_function(node_id=函数节点ID)` → 获取函数详情，包括 X0-X7 入口参数和 X0 返回值
5. `taint_analysis(from_specs=["reg:X0@入口行号"])` → 反向追踪输入数据来源（首页结果内联返回）
6. `get_tainted_lines` → 获取更多被污染的指令（如需翻页）
7. 综合分析，解释加密输入的数据来源（步骤 1-6 的数据采集是确定性的，第 7 步的解释是 AI 推理，复杂计算链的解释可能不完全准确）

**GUI 下的等效操作：** 菜单触发密码扫描 → 在结果中找到匹配位置 → 跳转到对应行 → 在调用树中找到所属函数 → 选中寄存器 → 右键打开污点配置 → 查看过滤结果。MCP 自动化了数据采集流程，但对结果的解读仍建议结合人工分析。

### 场景三：分析字符串和网络请求

**适用情况**：需要从 trace 中找到运行时生成的 URL、API 路径、密钥、token 等敏感字符串。

**提示词：**

```
打开 /path/to/trace.log，帮我分析字符串：

1. 提取所有运行时字符串，搜索包含 "http" "token" "key" "sign" 的字符串
2. 对找到的敏感字符串，用 search_instructions 搜索其地址附近的指令，找出是哪些指令在读写这些字符串
3. 对最关键的字符串（比如包含 URL 或 token 的），追踪其写入位置的数据来源
```

**AI 内部工具调用流程：**
1. `open_trace` → 获取基本信息
2. `get_strings(search="http")` → 搜索包含 http 的字符串
3. `get_strings(search="token")` → 搜索包含 token 的字符串
4. `search_instructions(query=字符串地址, addr_range=...)` → 搜索读写该字符串地址的指令
5. `get_trace_lines(start_seq=匹配行号, count=10)` → 查看引用上下文
6. `taint_analysis(from_specs=["mem:字符串地址@写入行号"])` → 追踪字符串数据来源（首页结果内联返回）
7. 综合分析，报告敏感字符串及其使用场景（步骤 1-6 的搜索数据是准确的，第 7 步的场景判断是 AI 推理）

**GUI 下的等效操作：** 切换到字符串面板 → 在搜索框输入关键词 → 右键 Show XRefs → 在弹出窗口中逐个点击跳转 → 再回来搜索下一个关键词 → 重复多次。MCP 可以一次性批量搜索多个关键词并自动关联数据，节省大量手动操作。

### 场景四：理解未知函数逻辑

**适用情况**：在 trace 中发现一个未知函数，需要理解它在做什么计算。

**提示词：**

```
打开 /path/to/trace.log，我需要分析一个函数：

1. 查看函数列表，找到地址为 0x12345 的函数（或搜索包含某特征的函数）
2. 获取这个函数的详细信息，包括入口参数和返回值
3. 对函数返回值（通常是 X0）做污点分析，追踪数据依赖
4. 查看函数内部调用了哪些子函数
```

**AI 内部工具调用流程：**
1. `open_trace` → 获取基本信息
2. `analyze_function` （无参数）→ 列出所有函数，找到目标
3. `analyze_function(node_id=节点ID)` → 获取函数详情（入口/出口行号、X0-X7 入口参数、X0 返回值、子调用列表）
4. `taint_analysis(from_specs=["reg:X0@出口行号"])` → 反向追踪返回值的数据来源（首页结果内联返回）
5. `get_tainted_lines` → 获取更多污点结果（如需翻页）
6. `get_call_tree(node_id=节点ID, depth=2)` → 查看子调用树（支持 1-3 层深度）
7. 综合分析，尝试用自然语言解释函数逻辑（步骤 1-6 的数据采集是确定性的；第 7 步是 AI 对结果的推理解释，简单计算逻辑通常准确，复杂的位运算/SIMD/多层嵌套计算可能需要人工验证）

**GUI 下的等效操作：** 在调用树中找到函数 → 双击跳转 → 查看寄存器面板 → 手动滚动到函数末尾 → 再看寄存器 → 右键构建依赖图 → 在 DAG 面板中自己分析节点关系。MCP 自动化了数据采集和初步分析，对简单函数逻辑的解释通常准确，但复杂计算仍建议结合 GUI 可视化和人工经验判断。

## 工具参考

### 开始分析

#### open_trace

打开 trace 文件并构建索引。**这是所有分析操作的第一步**，返回的 `session_id` 用于后续所有操作。当只有一个会话打开时，后续工具的 `session_id` 参数可省略。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| file_path | string | 是 | trace 文件的绝对路径 |
| force_rebuild | bool | 否（默认 false） | 即使缓存存在也强制重建索引 |
| skip_strings | bool | 否（默认 false） | 跳过字符串索引以加速打开 |

**返回**：session_id、module_name（模块名）、entry_address（入口地址）、trace_format（格式）、function_count（函数数量）、total_lines（总行数）、file_size（文件大小）、has_string_index（是否有字符串索引）、from_cache（是否从缓存加载）。

**注意**：首次打开大文件需要构建索引（千万行约 15 秒），构建完成后自动缓存，再次打开秒级加载。

### 浏览数据

#### get_trace_lines

获取指定范围的指令行。每行包含地址、反汇编、寄存器变化和内存访问信息。支持紧凑模式和完整模式。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| start_seq | u32 | 是 | 起始行号（从 0 开始） |
| count | u32 | 否（默认 20，最大 100） | 获取的行数 |
| full | bool | 否（默认 false） | 返回完整字段（含 raw、reg_before、so_offset、mem_size） |

**注意**：单次最多返回 100 行。需要查看更多行时多次调用，调整 start_seq 翻页。

#### get_memory

读取指定地址在指定指令行处的内存内容。显示执行到该指令时的字节值。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| address | string | 是 | 十六进制内存地址（如 `0xbffff000`） |
| seq | u32 | 否（默认为最后一行） | 行号（从 0 开始） |
| length | u32 | 否（默认 64，最大 256） | 读取的字节数 |

### 搜索

#### search_instructions

搜索 trace 中匹配文本或正则表达式的指令。支持按行号范围和地址范围过滤。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| query | string | 是 | 搜索内容。用 `/pattern/` 包装启用正则表达式 |
| use_regex | bool | 否（默认 false） | 使用正则表达式匹配 |
| case_sensitive | bool | 否（默认 false） | 大小写敏感 |
| max_results | u32 | 否（默认 30，最大 200） | 最大结果数 |
| seq_range | [u32, u32] | 否 | 限制搜索的行号范围，如 `[1000, 5000]` |
| addr_range | [string, string] | 否 | 限制搜索的地址范围，如 `["0x1000", "0x2000"]` |

**返回**：匹配的行号、地址、反汇编、寄存器变化，以及是否被截断。

**提示**：正则表达式示例 — `/bl.*0x[0-9a-f]+/` 可搜索所有带有特定地址的分支指令。

### 污点分析

#### taint_analysis

从指定寄存器或内存地址反向追踪数据依赖。**核心分析工具**。执行污点分析并**内联返回第一页结果**，无需额外调用即可查看初始结果。新的分析会自动覆盖之前的结果，无需手动清除。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| from_specs | string[] | 是 | 污点源列表，格式见下方 |
| data_only | bool | 否（默认 false） | 仅追踪数据依赖，忽略控制流 |
| addr_range | [string, string] | 否 | 限制分析的地址范围 |
| ignore_stack_ops | bool | 否（默认 true） | 过滤仅修改栈/帧指针寄存器（sp、x29）的行 |

**from_specs 格式：**
- `reg:X0@1234` — 追踪第 1234 行的寄存器 X0（寄存器名不区分大小写）
- `mem:0xbffff000@1234` — 追踪第 1234 行的内存地址
- `reg:X0@last` — 追踪寄存器 X0 最后一次被定义的位置

**返回**：被标记的指令数、总行数、百分比，以及第一页污点结果（内联）。如需更多结果，使用 `get_tainted_lines` 翻页。

#### get_tainted_lines

获取 `taint_analysis` 标记的被污染指令列表。支持分页和地址范围过滤。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| offset | u32 | 否（默认 0） | 分页偏移 |
| limit | u32 | 否（默认 50，最大 200） | 返回的最大行数 |
| addr_range | [string, string] | 否 | 限制返回结果的地址范围 |
| context_lines | u32 | 否 | 在每条污点行周围包含的非污点行数（提供上下文） |
| include_lines | bool | 否（默认 true） | 是否包含完整的指令行数据 |

### 函数分析

#### get_call_tree

获取函数调用树。支持指定展开深度，一次获取多层子节点。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| node_id | u32 | 是 | 节点 ID（从 0 开始，0 = 根节点） |
| depth | u32 | 否（默认 1，最大 3） | 展开深度（1 = 仅直接子节点，2-3 = 多层子节点） |

**返回**：节点数组（每个节点包含函数地址、名称、入口/出口行号、子节点 ID）、total_node_count（调用树总节点数）。

**提示**：从 `node_id=0` 开始探索，获取顶层函数，再逐层展开子节点。使用 `depth=2` 或 `depth=3` 可减少调用次数。

#### analyze_function

多功能函数分析工具，支持三种模式：

**模式一：按 node_id 查询函数详情**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| node_id | u32 | 是 | 调用树节点 ID |

**返回**：函数地址、名称、入口/出口行号、行数、X0-X7 入口参数值、X0 返回值、子调用列表。

**模式二：按函数名搜索**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| func_name | string | 是 | 函数名（支持模糊搜索） |

**返回**：匹配的函数列表。

**模式三：列出所有函数（无额外参数）**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| offset | u32 | 否（默认 0） | 分页偏移 |
| limit | u32 | 否（默认 50） | 返回的最大数量 |

**返回**：按函数名分组的调用列表，包含每次调用的行号。适合快速了解哪些函数被调用最多。

### 字符串分析

#### get_strings

列出 trace 中提取的运行时字符串。支持搜索和过滤。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| min_len | u32 | 否（默认 4） | 最小字符串长度 |
| search | string | 否 | 搜索包含此子串的字符串 |
| offset | u32 | 否（默认 0） | 分页偏移 |
| limit | u32 | 否（默认 50，最大 200） | 返回的最大数量 |

**返回**：字符串数组（地址、内容、编码、长度、交叉引用数）、总数、是否有更多。

### 密码识别

#### analyze_crypto

扫描 trace 中的密码算法魔数常量，并**返回每个匹配位置的上下文指令**。覆盖 AES、SHA256、MD5 等 28 种模式。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| session_id | string | 否 | 会话 ID（单会话时可省略） |
| context_lines | u32 | 否 | 每个匹配位置前后包含的上下文指令行数 |

**返回**：检测到的算法名称、匹配的魔数值、出现位置的指令地址和汇编内容、上下文指令。结果会自动缓存。

## 限制与约束

### 数据量限制

| 限制 | 值 | 说明 |
|------|-----|------|
| 单次获取行数 | 最多 100 行 | `get_trace_lines` 的 count 上限 |
| 搜索结果数 | 最多 200 条 | `search_instructions` 的 max_results 上限 |
| 内存读取长度 | 最多 256 字节 | `get_memory` 的 length 上限 |
| 污点结果分页 | 最多 200 行 | `get_tainted_lines` 的 limit 上限 |
| 字符串分页 | 最多 200 条 | `get_strings` 的 limit 上限 |

超出限制时使用分页参数（offset/limit）获取更多数据。

### 工具调用顺序

- 所有操作必须先调用 `open_trace` 获取 session_id
- 当只有一个会话打开时，后续工具的 `session_id` 参数可省略
- `taint_analysis` 执行分析并内联返回第一页结果，无需再单独调用 `get_tainted_lines` 查看初始结果
- 新的 `taint_analysis` 调用会自动覆盖之前的污点结果，无需手动清除
- `get_tainted_lines` 用于翻页获取更多污点结果，或使用 addr_range/context_lines 等高级过滤
- `get_strings` 需要字符串索引（默认在 open_trace 时构建，可通过 skip_strings 跳过）
